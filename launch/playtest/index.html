PLAYTEST LOOP

PUBLIC PLAYERS -> APPOINTED REVIEWERS -> OWNER APPROVAL

This repo is a public preview. Use this loop to get builds into community hands, turn playtest notes into reviewable GitHub issues, and let appointed volunteers recommend which changes are ready for the next public slice.

LANES

PLAYERS

Play and report

Play the public demo, then file one issue per bug, balance note, or usability problem.

Best reports include: game, browser/device, steps to reproduce, expected result, and actual result.

VOLUNTEERS

Triage and recommend

Community members can apply for a narrow review scope. Appointed reviewers can reproduce, label, and recommend approve, needs-work, or defer.

OWNER

Merge and publish

Maintainers keep final authority for code changes, default-branch merges, itch uploads, Kickstarter copy, and public deploys.

VOLUNTEER INTAKE

Need more people in the loop? Use the recruitment kit for ready-to-send posts and PA screening rules.

  1. Self-qualify publicly. Applicant fills out the volunteer application form with a GitHub handle, preferred scope, test environment, and sample review.
  2. PA scope check. The PA or maintainer checks that the applicant chose one narrow lane and acknowledged the safety rules.
  3. No-access pilot. Applicant leaves 1-3 review recommendations as public issue comments or volunteer review issues. No repository write access is needed.
  4. Appoint if useful. If the pilot is clear, kind, scoped, and safe, Daniel/maintainer records the scope and may grant issue-triage access.
  5. Review access later. Stale, unsafe, off-scope, or disruptive volunteers lose reviewer status immediately.

LOOP

  1. Publish a playable slice. Link the build from the support hub and pin known limitations.
  2. Collect structured issues. Playtesters use the feedback template instead of free-form DMs.
  3. Volunteer review. Appointed reviewers reproduce the issue or inspect the proposed change and record a recommendation issue/comment.
  4. Owner gate. Daniel or a maintainer accepts the recommendation, schedules the work, or closes it with a reason.
  5. Ship notes. When a new preview is published, link the shipped issues and ask for focused retests.

ACCESS MODEL

RoleAllowedNot allowed
Playtester Open issues, add screenshots, confirm whether a fix works. Approve merges, request secrets, handle private player data.
Volunteer reviewer Reproduce reports, recommend priority, request clearer evidence, mark approval status. Bypass governance, publish builds, speak for the project owner.
Maintainer / owner Assign volunteers, merge PRs, publish builds, approve external announcements. Delegate final accountability for data, money, minors, or public brand claims.

Safety rule: do not ask playtesters for real names, age, payment details, home addresses, or private medical/identity information. If a report includes accidental personal data, hide or delete the public copy and ask for a scrubbed version.

ABUSE-RESISTANT DEFAULTS

RiskDefault gatePA / maintainer action
Spam or low-effort applicants Application requires a sample review, scope, and acknowledgements. Close with no access; invite a better sample later if appropriate.
Bad actor seeking control Volunteer status starts with public comments only, not write access. Never grant secrets, billing, deploy, or merge rights through this loop.
Off-scope arguments Each reviewer has one named scope and one recommendation form. Ignore or close off-scope reviews; narrow the assignment before continuing.
Private data leakage All forms warn against personal data and ask for public-only details. Remove unsafe public content and ask for a scrubbed report.

VOLUNTEER APPOINTMENT CHECKLIST

  1. Daniel or the PA links the accepted volunteer application.
  2. Daniel names the volunteer and scope: game, issue category, or release slice.
  3. Maintainer grants the least GitHub access needed, usually no access for the pilot and issue triage only after trust is earned.
  4. Volunteer reviews with the public template and does not request secrets, private builds, or player identity details.
  5. Owner can remove reviewer status at any time; stale access gets reviewed during governance audits.
